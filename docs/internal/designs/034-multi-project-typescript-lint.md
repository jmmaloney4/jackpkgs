# ADR-034: Multi-Project TypeScript Lint in `just lint`

## Status

Proposed

## Context

- The `just lint` recipe (generated by `just.nix`) runs `tsc --noEmit` once against the repo root, guarded by `if [ -f tsconfig.json ]`.
- The `checks.nix` module already supports `jackpkgs.checks.typescript.tsc.packages` — a list of workspace-relative paths to type-check in Nix sandbox builds. But the **just lint recipe ignores this list** and only checks the root tsconfig.
- Monorepos with divergent TypeScript projects (e.g. Pulumi stacks with `nodenext` module resolution + a Vite frontend with `bundler` resolution) cannot share a single root tsconfig. A root config that covers one breaks the other.
- Current workaround: consumers create a root `tsconfig.json` covering one group (e.g. `deploy/**/*.ts`) and handle the other group manually (local justfile recipe or CI step). This means `just lint` doesn't check everything it should.

## Decision

Extend the just lint recipe to iterate over `jackpkgs.checks.typescript.tsc.packages` (or a new dedicated option), running `tsc --noEmit --project <path>/tsconfig.json` for each entry. When `packages` is null (unset), fall back to the current single-root behavior (`if [ -f tsconfig.json ]; then tsc --noEmit`).

### Option A — Reuse `checks.typescript.tsc.packages` (RECOMMENDED)

Use the existing `packages` option from the checks module to drive the lint recipe:

```nix
jackpkgs.checks.typescript.tsc.packages = [ "." "services/lens/frontend" ];
```

Generated lint recipe:

```bash
# tsc (TypeScript type checker)
for _tsc_project in . services/lens/frontend; do
  if [ -f "$_tsc_project/tsconfig.json" ]; then
    printf '%s\n' "==> tsc ($_tsc_project)"
    tsc --noEmit --project "$_tsc_project/tsconfig.json"
  fi
done
```

When `packages` is null, fall back to the current behavior:

```bash
# tsc (TypeScript type checker)
if [ -f tsconfig.json ]; then
  printf '%s\n' "==> tsc"
  tsc --noEmit
fi
```

### Option B — New `just.typescript.projects` option

Add a separate option under the just module that only affects the lint recipe. Decouples lint from `checks.nix` but introduces a second source of truth for the same information.

### Option C — Root tsconfig with project references

Require consumers to create a root tsconfig.json using TypeScript project references (`"references": [{"path": "..."}]`) and run a single `tsc --build` from root.

Pros: single invocation, TypeScript-native.
Cons: requires `composite: true` in every sub-project's tsconfig; doesn't work when sub-projects have incompatible compiler options (the whole reason for this ADR); significant consumer setup burden.

## Consequences

### Benefits

- `just lint` type-checks all TS projects in a monorepo, not just the root.
- Consumers with a single root tsconfig (the common case) see zero change — `packages = null` falls back to current behavior.
- Reuses an existing option (`checks.typescript.tsc.packages`) rather than introducing a new one.
- No changes required to `checks.nix` — only the lint recipe generation in `just.nix`.

### Trade-offs

- The lint recipe and the Nix check derivation now share configuration via `packages`, but execute differently (lint runs `tsc --project` per entry; the Nix check runs a single `tsc --noEmit` from a sandboxed copy). This divergence already exists and is acceptable — the lint recipe runs in the devshell with access to the real filesystem, while the Nix check runs in a sandbox.
- `packages = ["."]` would run `tsc --noEmit --project ./tsconfig.json`, which is functionally identical to `tsc --noEmit` from root. No behavior change for single-project repos that explicitly set packages.

### Risks & Mitigations

- **Risk: `packages` is null by default (auto-discovery from pnpm-workspace.yaml).** Auto-discovery returns workspace package paths, not tsconfig project roots. Mitigation: only iterate when `packages` is explicitly non-null. When null, fall back to root-only behavior. This matches consumer expectations — if you want multi-project lint, you must explicitly list the projects.
- **Risk: `extraArgs` applies globally to all projects.** Some projects might need different flags. Mitigation: not in scope for this ADR. `extraArgs` is already global in the checks module. Per-project args can be addressed later if needed.

## Implementation Plan

1. In `just.nix`, read `checksCfgForRecipes.typescript.tsc.packages` when generating the lint recipe.
2. When `packages` is a non-empty list, generate a for-loop iterating over the entries.
3. When `packages` is null, generate the current `if [ -f tsconfig.json ]` block unchanged.
4. Add nix-unit tests in `tests/lint-recipe.nix` for both paths (single-root and multi-project).
5. Verify backward compatibility: no change to generated lint recipe when `packages` is unset.
